# fm 画面描画アーキテクチャ 方針まとめ

## 1. 背景・経緯

FDライクな罫線ヘッダー、段組、2ペイン構成を実現しようとする中で、既存の
「テンプレート文字列1枚で画面全体をWYSIWYGに表現する」設計（`#name####`の
タグ記法、正規表現パターンマッチ案など）を拡張し続けると、以下の問題に
ぶつかった。

- 罫線の伸縮など、データを持たない装飾要素の表現に無理が生じる
- 2ペイン（独立したデータソースが複数）と段組（同一データソースの折り返し）を
  同じ記法で扱おうとすると設計が複雑化する
- 画面幅変化時の再配置（ブレークポイント、Rect分割）を持ち込むとさらに複雑化する
- 記法を拡張するたびに「現状の実装に構造を引っ張られる」状態になった

そこで、実装方式の議論から一度離れ、アプリケーション層の要件定義からやり直した。

### コンセプトの再確認

fmの「試作・変更のしやすさ」というコンセプトは、特定の実装手段
（テンプレート文字列によるWYSIWYG記法）を指すものではない。

本質的な要求は以下：

- 文字列を領域に収める計算（幅・位置・切り詰め）を毎回自分で書かずに済むこと
- 試行錯誤のサイクルが速いこと

この要求を満たす手段は、必ずしも「1枚のテンプレート文字列」である必要はない。

> **後日の追記**: 検討の結果、「本質的な目的にはWYSIWYG性は別に必須ではなかった」
> ことが確認された。当初の想定より難しいことが判明したための再定義であり、
> `fm-mode-interface-design.md`・`fm-interface-design.md`はこの前提の上に
> 積み上げられている。

### 責任範囲を判断する基準

**「もしGUIに移植するとしたら、この境界線のままで自然か」を判断基準とする。**
（実際にGUI対応する予定はないが、TUI固有の関心事と、ファイルマネージャ本来の
関心事を切り分けるための思考実験として採用する。）

この基準で分類すると：

| 分類 | GUIでも同じ概念が存在するか | 備考 |
|---|---|---|
| ファイルシステム・コンピュータの情報 | 存在する | TUI/GUIに依存しない共通データ |
| ファイルマネージャとしての情報（状態） | 存在する | 選択状態・操作モード・通知などGUIにも共通 |
| 表示に関する情報 | 存在しない（TUI固有） | GUIなら「ウィンドウサイズ」「フォント」等に置き換わる |

「表示に関する情報」こそがTUI固有の関心事であり、画面描画処理の責任は、
この情報を使って他の2カテゴリ（普遍的なデータ）を画面に落とし込むこと。
TUI特有の計算処理（幅計算・レイアウト）をここに閉じ込めておけば、
将来的な差し替えが局所化される。

## 2. 全体構造の方針（確定事項）

1. **ヘルパー関数群**が、文字列を領域サイズに適応させる処理（幅指定・寄せ・
   罫線生成・繰り返し展開など）を提供する。1枚のテンプレート文字列という
   制約を持たない。
2. **モード（画面）ごとに、描画処理とキー処理を1組にしたまとまり**を用意する。
   同じキー入力でも、どのモードにいるかによって解釈が変わる（例: 通常モード
   の`j`はカーソル移動だが、検索モードの`j`は検索文字列への入力になる）ため、
   描画処理だけでなくキー処理（状態変更ロジック）もモードごとに切り替える
   必要がある。「1画面＝1つの描画処理」ではなく「1モード＝1つの描画処理＋
   1つのキー処理」を基本単位とする。
3. **モードの切り替え**は、状態（今どのモードがアクティブか）を持つ形で
   表現する。将来的に確認ダイアログや検索モードが増えても、同じ枠組みで
   対応できる。
4. **2ペイン・段組・可変レイアウト**は、いずれも「モードごとの描画処理」の
   中で、通常のLuaのコード（条件分岐・関数呼び出し）として素朴に表現する。
   特別な記法を新設しない。

> **後日の追記**: 「モード」という言葉は`fm-mode-interface-design.md`の
> 検討で「スクリーン」（画面バリエーション）と「インボーカー」（コマンド
> 実行の統一入口）という2つの概念に分かれることが判明した。詳細は
> `fm-interface-design.md`を参照。

### 今回手放すもの

- テンプレート文字列（`#name####`、正規表現パターンなど）でレイアウト全体を
  1枚に表現する設計
- `columns` によるタグ⇔データの対応定義という仕組み

### 今回残すもの

- UTF-8表示幅計算（既存の `utf8width` モジュール。Rust移植も検討中）
- `screen.write()`、`fs.*` などRust側の薄いインタフェース層

## 3. モードの設計方針（確定事項）

### モードの構造

状態遷移を一から書き下すのは「試作のしやすさ」というコンセプトに反する。
新しいモードを追加するたびに遷移表全体を見直す必要があるようでは、
テンプレート記法の拡張で身動きが取れなくなった経緯と同じ罠にはまる。

ファイルマネージャとして実際に必要になりそうなモード（検索、確認ダイア
ログ、コマンド入力など）を洗い出すと、いずれも「通常モードの上に一時的に
乗って、完了したら通常モードに戻るだけ」という共通の性質を持つ。検索
モードから確認ダイアログへ直接遷移するといった、複雑な組み合わせは
実際には想定しなくてよい。

そこで以下を方針とする。

- **通常モード（ファイル一覧表示）を唯一の基本状態とする**
- それ以外のモードは、通常モードに一時的にかぶせるだけの「割り込み」として
  扱う。割り込みが終了したら、必ず通常モードに戻る
- この前提により、モード同士の複雑な組み合わせ・遷移表を考える必要が
  なくなる。「今、一時的な割り込みが乗っているか、いないか」に近い単純な
  構造で足りる

> **後日の追記**: このスター型（必ず通常モードに戻る）の前提は、
> `fm-mode-interface-design.md`の検討で「1列表示⇔2段組のような、戻った先の
> 描画処理そのものが変わるケース」に対応できないことが判明し、崩れた。
> 「モードは状態ではなく、メインループから呼ばれる処理である」という
> 捉え直しにより解消している。詳細は`fm-mode-interface-design.md`の
> 「重要な発見」を参照。

### 表示切り替えとモードの関係

「表示切り替え」とは、ペインの選択（Tabキーなどで操作対象のペインを移す）
や、詳細表示／簡易表示の切り替えなど、表示上のバリエーション全般を指す
（ペインに限定した概念ではない）。これとモード切り替え（検索・確認
ダイアログに入る）は、性質の異なる独立した軸として扱う。

- **表示切り替え**: 「どのデータを、どう見せるか」を選ぶ。通常モードの
  中で完結する話であり、モードの一種として統一する必要はない
- **モード切り替え**: 「今どんな入力の受け付け方をしているか」を選ぶ。
  表示切り替えとは独立して存在する（検索モード中でも、表示上のバリ
  エーション自体は維持される）

この2つを独立した軸として分離しておくことで、表示上の設定（ペインの
ソート順、フィルタなど）を増やしてもモードの複雑さには影響せず、モードを
増やしても表示切り替えの構造には影響しない。「機能が増えれば必要な
データや処理もある程度増えるのは避けられない」という前提のもと、増える
箇所が予測可能な形で局所化されているかを設計の判断基準とする。

表示切り替え自体は、モードのような「割り込み→復帰」の構造を持つ必要が
ない。モードの状態遷移は「通常モードを中心としたスター型」（通常モード⇔
検索/確認ダイアログなど、必ず通常モードに戻る）である一方、表示切り替え
は「次にどの表示へ切り替えるか」だけを気にすればよく、「元に戻る」という
概念自体が存在しない。そのため、表示切り替えはモードとは別の、もっと
単純なサブの状態（例えば`display_style`のような値が"detail"→"simple"→
"detail"のように巡回的に変わるだけの、フラットな状態）として実装してよい。

### 拡張の方針（モードの追加のしやすさ）

モードのインタフェース設計は、全てのユースケースを事前に網羅して完璧な
設計を決め切ることはできない。考えれば方針が出る他の論点と異なり、実際に
作ってみないとわからない部分が残る種類の問題である。

そのため、以下の進め方を採る。

- **「厳密なインタフェース仕様」ではなく「試作を始めるための最初の型」だけ
  決める**。設計の変更・追加が発生しても基本が崩れない程度のユースケース
  を想定する
- 最初の型としては、各モードが「描画」と「キー処理」の2つの関数を持つ、
  という最低限の形を仮決めする
- 具体的な引数の数・形式・エラーの返し方などは決め切らず、実際に最小構成
  （ファイル名をただ並べるだけの簡易的なファイル一覧表示、キーが押されたら
  何らかのコマンドを実行するだけの単純なモード）を1つ動かしてみて確認する
- 動かしてみて書きにくければインタフェースを直す。うまくいけば正式な方針
  として採用する

これは、fmのRustフレームワークを最初に作った際に「空の`on_init`/`on_key`
が呼べる」という最小構成からまず動かし、そこから肉付けしていった進め方と
同じアプローチである。

## 4. データ構造の方針

### 画面描画処理に渡すデータ（3カテゴリ）

**1. ファイルシステム・コンピュータの情報**

- カレントディレクトリのパス
- ファイル一覧（ファイル名、サイズ、更新日時、パーミッション、ディレクトリかどうか）
- ドライブ／ストレージの空き容量・使用量
- マシン名・ユーザー名

**2. 表示に関する情報**

- ターミナルの幅・高さ
- カラー表示に対応しているか
- 表示言語設定
- （文字幅計算に関わる）ロケール・エンコーディング情報

**3. ファイルマネージャとしての情報**

- カーソル位置
- マーク（複数選択）されているファイル
- ペイン数、どのペインがアクティブか
- 現在のモード（通常操作／検索中／確認ダイアログ中など）
- 直前の操作に対する通知メッセージ（コピー完了、エラーなど）
- ソート順・フィルタ状態（隠しファイル表示など）

### 状態の所有権（確定事項）

ある状態をペインごとに持つべきか、アプリ全体で1つ持つべきかは、以下を
判断基準とする。

**その状態は、特定のディレクトリ（ペイン）に紐づく操作の結果か、それとも
ディレクトリに関係なくアプリ全体に関わるものか**

- **ペインに紐づく（ペインごとに持つ）**: カーソル位置、マーク、検索
  文字列、ソート順、フィルタ設定など。そのディレクトリに対する操作は、
  そのディレクトリをパスに持つペインに属するデータである
- **アプリ全体（グローバルに持つ）**: 今アクティブなのはどのペインか、
  確認ダイアログの内容など。特定のペインの操作結果を跨いで表示される
  一時的なものはグローバルな状態として扱う

## 5. 処理順序（確定事項）

1画面分の更新は、以下の4ステップで構成する。

1. **ユーザー入力**
2. **状態変更** — 現在アクティブなモードのキー処理に委譲し、マーク処理・
   カーソル移動・ソート順変更・検索文字列の編集など、キー入力に応じた
   処理を行う。マーク処理とカーソル移動を別ステップに分ける理由はない
   （どちらも「3. ファイルマネージャとしての情報」を書き換えるという
   同じ性質の操作）ため、1つのステップに統合する。複数の変更が1回の
   キー入力に紐づく場合（例:「マークして1つ下に移動」）は、このステップ
   内で順に実行する。同じキーでもモードによって処理内容が変わる
   （「2. 全体構造の方針」を参照）。
3. **データ準備** — ソート・フィルタ等を適用し、表示用に加工済みのデータを
   構築する（未決事項のソート・フィルタ処理はここに位置づけられる）。
4. **画面描画** — 現在アクティブなモードの描画処理に、データ準備の結果を
   渡して描画する。

この4ステップに統一しておくことで、将来的に操作（ファイルコピー・削除、
検索モードへの切り替えなど）を追加する際も「状態変更ステップに処理を足す」
という一貫した置き場所が決まる。ステップ2・4がモードごとに切り替わる点を
除けば、ステップ自体の構成はモードによらず共通である。

なお、コマンド実行（`less`や外部エディタの起動など）に伴う画面描画は
この一連の流れとは別に発生しうるが、今回の整理では一旦対象外とする。

## 6. エラー・異常系の扱い（確定事項）

外部コマンド（`less`、`xxd`など）の実行失敗は考慮しない。実行中は
ターミナルの制御がそのコマンドに渡っており、コマンド自身が自分の画面に
エラーを表示し、ユーザーがキーを押せば終了する。fm自身の状態やモードには
影響しない。

「fmとして扱うべきエラー」は、fm自身が画面上で表示する必要があるもの
（`fs.list()`の失敗など、ディレクトリが開けない・権限がない等）に限られる。

エラーメッセージの表示は、ファイル一覧などの通常の画面描画ロジックには
混ぜ込まず、独立した仕組みとして扱う。具体的な見せ方（一時的に表示して
キー押下で消える方式か、あらかじめ決められたメッセージエリアに表示して
次のコマンド実行まで残る方式か）は実装時の設計次第とし、今は決め切らない。
いずれの見せ方であっても、画面描画の状態遷移やモードの状態遷移に
含めるほどのものではない、と判断する。

## 7. 未決事項（次回検討）

- ソート順・隠しファイル表示フィルタの扱い
  - 設定・状態自体は「4. データ構造の方針」の「3. ファイルマネージャ
    としての情報」に属する
  - 実際の並び替え・絞り込み処理（生のファイル一覧 → 表示用に加工済みの
    一覧）は表示とは無関係なデータ加工処理であり、画面描画処理とは別の
    「データ準備層」に置くべきという方向性まで話が出ている（未確定）
- モードごとの描画処理・キー処理インタフェースの具体的なシグネチャ
  （引数の数・形式、モジュールとしての持たせ方）
- 3カテゴリのデータをモードの描画処理・キー処理にどう渡すか（1つの引数に
  まとめるか、複数引数に分けるか）
- モード切り替えの具体的な表現方法（状態の持ち方、切り替えのトリガー）
- ヘルパー関数群（`ui.*`相当）の具体的なAPI
