# fm モード・インタフェース設計 検討ノート

## 導入

以下のユースケースを考えること。

- ファイル一覧
- ファイル一覧の別バリエーションの表示に変更
- ファイル操作コマンド実行（ディレクトリ作成、ファイル削除など）
- コマンド正常終了後、ファイル一覧に戻る
- コマンド異常終了後、ファイル一覧に戻る
- ファイルの表示を変更（ソート、フィルタ、検索など）
- 外部コマンド実行→正常終了
- 外部コマンド失敗→異常終了

これらすべてに対して個別に設計を詰めるのではなく、性質が明らかに異なる
代表的な4つのケースをまず検討する。それ以外のユースケースが、この4つの
バリエーションとして説明できるかどうかを後から確認する。

代表的な4つのケース：

1. **入力を伴わない、即座に完結する操作**
   （例: 確認ダイアログを伴うファイル削除）
2. **入力を伴う、継続的な操作**
   （例: 検索）
3. **状態変更を伴わない、見た目だけの切り替え**
   （例: 表示バリエーションの変更）
4. **fmが一時的に制御を手放す操作**
   （例: 外部コマンド実行）

### 検討の流れ

各ケースについて、以下の順で検討する。

1. **ユーザーから見た処理の流れを作る**（画面とキー入力）
2. **処理単位に分割**
3. **input / output を定義**
4. **インタフェースを定義**

この順序を採る理由は、記法やAPIの都合を先に考えてしまうと、今回の
テンプレート記法の拡張がそうであったように「現状の実装に構造を引っ張られ
続ける」状態に陥りやすいため。ユーザー体験の具体から出発し、最後に構造へ
落とし込む向きで進める。

### 検討する4ケース（進捗）

1. 入力を伴わない、即座に完結する操作 → 確認ダイアログを伴うファイル削除
2. 入力を伴う、継続的な操作 → 検索
3. 状態変更を伴わない、見た目だけの切り替え → 表示バリエーションの変更
4. fmが一時的に制御を手放す操作 → 外部コマンド実行（`less`など）

各ケースはステップ1〜3まで検討し、4ケースすべて終わってから
インタフェース定義（検討の流れのステップ4）をまとめて行う。

> **読み方の注意**: 以下のケース1〜3は、検討の過程では「コマンド実行
> モードへ遷移する」という状態遷移の言葉で記述されていた。しかし4ケース
> すべての検討を終えた後、「モードは複数回のメインループ反復にまたがって
> 保持され続ける状態ではなく、メインループの中で呼ばれて仕事をして戻る
> だけの処理（関数）である」という重要な理解の見直しがあった（詳細は
> 本ノート末尾「重要な発見」を参照）。そのため、ケース1〜3のinput/output
> （ステップ3）は、この新しい理解に基づいて書き直し済みである。ステップ
> 1・2の記述は検討過程の記録として、遷移という言葉を残したまま維持する。

## コマンド実行フロー（基本パターン）

代表ケース「1. 入力を伴わない、即座に完結する操作」（確認ダイアログを
伴うファイル削除）を検討する過程で見えてきた、ファイルマネージャの
一般的な機能に共通するフロー。以下の4つの代表ケース（確認ダイアログ、
検索、外部コマンド実行、表示バリエーションの切り替え）は、いずれも
このパターンの一部ステップが省略されたバリエーションとして説明できる。

コマンド実行はファイル削除・ファイル検索・外部コマンド実行など、
あらゆるコマンドを想定する。コマンドによっては、そのまま操作を終える
もの（削除）もあれば、内部データを操作してそれがファイル一覧に反映
されるもの（検索によるフィルタ）もある。

1. **コマンド実行のキー操作**（例: `d`、`/`、`enter`、`j`）
2. **確認 or 追加入力**（option）（例: 削除確認の`y`/`n`、検索文字列の入力）
3. **コマンド実行**（例: 削除処理、フィルタ処理、`fs.run("less ...")`、
   カーソル移動処理）
4. **実行成否表示**（option）（例: 削除失敗時のエラー表示。成功時は
   ファイル一覧の変化そのものが結果を示すため、通知は不要な場合が多い）
5. **ファイル一覧表示**（更新後の一覧、絞り込み後の一覧、カーソル移動後の
   一覧など）

具体例を並べると、各ステップの有無の違いだけであることが分かる。

| ステップ | 削除 | 検索 | 外部コマンド実行(`less`) | 単純なカーソル移動 |
|---|---|---|---|---|
| コマンド実行のキー操作 | `d` | `/` | `enter`(拡張子なしファイル) | `j` |
| 確認 or 追加入力(option) | 確認(y/n) | 検索文字列の入力 | なし | なし |
| コマンド実行 | 削除処理 | フィルタ処理 | `fs.run("less ...")` | カーソル移動処理 |
| 実行成否表示(option) | 失敗時のみ | なし | なし(コマンド自身が表示) | なし |
| ファイル一覧表示 | 更新後の一覧 | 絞り込み後の一覧 | 元の一覧 | カーソル移動後の一覧 |

### このパターンで説明できる機能

- ファイルのコピー・移動・リネーム（追加入力: コピー先/新しい名前 →
  コマンド実行 → 成否表示: 失敗時のみ）
- ディレクトリ作成（追加入力: ディレクトリ名 → コマンド実行 → 成否表示）
- ファイル圧縮・解凍（確認 or 追加入力 → コマンド実行 → 成否表示）
- ソート順変更（確認・追加入力なし → コマンド実行: ソートキーの切り替え
  → 一覧表示のみ）
- マーク（トグル）（確認・追加入力なし → コマンド実行: マーク状態を反転
  → 一覧表示のみ）
- 複数ファイルの一括操作（マークしたファイルを全部削除、など）
  ─ 「コマンド実行」の中身が単発ではなくループになるだけで、パターン
  自体は壊れない

### このパターンに収まりにくいもの（要注意）

- **プレビュー機能**（カーソルを動かすたびに、隣にファイルの中身を表示）
  ─ 「コマンド実行」ではなく「表示バリエーション」に近い。カーソル移動の
  結果としてプレビュー内容が変わるだけなら、「ファイル一覧表示」ステップ
  の中身が変わる話として収まりそう
- **タブ補完**（リネーム時に既存ファイル名を補完） ─ 「追加入力」ステップ
  の中がさらに対話的になるケース。この5ステップの粒度では表現しきれず、
  「追加入力」ステップ自体がもう1段階の別パターンを持つ、入れ子構造に
  なる可能性がある

### 位置づけ

このコマンド実行フローを基本パターンとし、以降の代表ケースの検討は、
このパターンの各ステップを具体化していく形で進める。

インタフェース定義（検討の流れのステップ4）は、4つの代表ケースすべての
検討が終わってからまとめて行う。個々のケースではステップ1〜3（処理の
流れ、処理単位の分割、input/output定義）までを進める。

## ケース1: 入力を伴わない、即座に完結する操作（確認ダイアログを伴うファイル削除）

コマンド実行フロー（基本パターン）を前提に検討する。

### ステップ1: ユーザーから見た処理の流れ

1. **コマンド実行のキー操作**: 通常モードでカーソルを合わせて`d`を押す
2. **確認 or 追加入力**: 「本当に削除しますか？(y/n)」が、ファイル一覧は
   そのまま表示された状態で画面に重ねて表示される。`y`/`n`の入力を待つ
3. **コマンド実行**: `y`が押されたら削除処理を実行
4. **実行成否表示**: 成功時は何も表示しない（ファイル一覧の変化が結果を
   示す）。失敗時はエラーメッセージを表示
5. **ファイル一覧表示**: 成功時は更新後の一覧、失敗時（`n`だった場合も
   含む）は元の一覧

この5ステップは1回のキー入力で一気に進むわけではなく、ステップ2の途中で
ユーザーの入力（`y`/`n`）を待つために一旦止まる。

### ステップ2: 処理単位に分割

ステップ2で一旦止まるという性質から、この一連の処理は「モード」として
扱う。ただし、このモードは「確認モード」のような限定的な名前にせず、
**「コマンド実行モード」**として、確認・追加入力・実行・成否表示という
コマンド実行フロー全体を担う汎用的なモードと位置づける（削除以外の
検索や他のコマンドも、将来的に同じ枠組みに乗ることを想定）。

| フェーズ | どのモードで発生するか | 内容 |
|---|---|---|
| 1. コマンド実行のキー操作(`d`) | 通常モード | コマンド実行モードへ遷移。何のコマンドか（削除）を保持 |
| 2. 確認(y/n) | コマンド実行モード | ユーザー入力を待つ（このモードの`on_key`が担当） |
| 3. コマンド実行 | コマンド実行モード内の`y`処理 | 削除処理を実行 |
| 4. 実行成否表示 | コマンド実行モード内の`y`処理の結果 | 失敗時はメッセージをセット |
| 5. ファイル一覧表示 + 通常モードへ復帰 | コマンド実行モード内の`y`/`n`処理 | 通常モードへ遷移 |

なお、「データ準備」ステップ（ソート・フィルタを適用して表示用データを
作る）を、モード固有の処理にするか、モードに依らない共通処理にするかは
まだ決めない（保留）。

### ステップ3: input / output を定義

削除のケースに限定して定義する。「重要な発見」（本ノート末尾参照）を
踏まえ、「コマンド実行モードへの遷移」ではなく「コマンド実行処理の
呼び出し」として記述する。呼び出しの間だけ存在すればよいデータは、
状態として保持し続ける必要がなく、呼び出しのローカルな引数・戻り値と
して扱える。

**通常モードの`on_key`が、コマンド実行処理を呼ぶときに渡すデータ
（削除の場合）**

コマンド実行処理の入り口は1つに統一し、何のコマンドかを識別する
コマンド名を必ず渡す（詳細は「重要な発見」の後、統一入口の検討を参照）。

| 項目 | 内容 |
|---|---|
| コマンド名 | `"delete"` |
| 対象ファイル | 削除しようとしているファイルの情報（パス、名前など） |
| 確認メッセージ文言 | `"本当に削除しますか？"`（または呼び出し側が生成して渡す） |

**削除コマンドの処理（呼び出しの中で、`y`/`n`が来るまでブロッキングして
待ちつつ完結する）**

| 処理 | 内容 |
|---|---|
| 確認メッセージの重ね書き | `y`/`n`を待つ間、画面に確認メッセージを表示し続ける |
| `y`が押された場合 | 削除処理を実行し、成否を判定する |
| `n`が押された場合 | 何もしない |

**削除コマンドの処理からの戻り値**

| 項目 | 内容 |
|---|---|
| エラーメッセージ（あれば） | 失敗時のみ。呼び出し元（通常モード）の`message`にセットするための文字列 |

呼び出しが終わって`on_key`に処理が戻ったら、`draw()`が呼ばれて通常モード
の描画が行われる。「コマンド実行モードが保持する状態」「コマンド実行
モードの`draw`」という、状態として持続するモード特有の概念は不要になる
（確認メッセージの重ね書きは、呼び出しの中のブロッキングループが自分で
画面に書く、という処理として完結する）。

> **後日の追記**: この「ブロッキングループが自分で画面に書く」という
> 実現方法は、`view`が唯一の描画経路であるという原則を崩すという懸念が
> 指摘され、`fm-interface-design.md`で「確認ダイアログもスクリーンの
> 一種として表現する」方式に置き換えられた。詳細は同ファイルの
> 「確認ダイアログの実現方法（ブロッキングループを使わない）」を参照。

## ケース2: 入力を伴う、継続的な操作（検索）

コマンド実行フロー（基本パターン）を前提に検討する。当初は「1文字ごとに
リアルタイムで絞り込む」形も検討したが、まずは削除ケースと同じ「確定して
から実行する」型（`Enter`で確定するまで絞り込みは実行されない）で進める。
リアルタイム絞り込みは、必要になった時点で改めて拡張を検討する。

> **後日の追記**: リアルタイム検索を見送った判断について、「全ての
> ファイラーに搭載されている機能ではなく、『試作・変更がしやすい』という
> 現在の目的にも必須ではない」ことを理由に妥当と確認済み。

### ステップ1: ユーザーから見た処理の流れ

1. **コマンド実行のキー操作**: 通常モードで`/`を押す
2. **追加入力**: `on_key`の中で`terminal.read_line(x, y, max_width, "/")`
   を呼ぶ。Rust側がEnter/Escapeまで入力を監視し、確定した文字列
   （またはキャンセル）を返す
3. **コマンド実行**: 確定した文字列でフィルタ処理を実行し、検索状態を
   セットする
4. **通常モードへ復帰**: ファイル一覧の表示は、通常モードへ戻った後の
   `draw`（データ準備→画面描画）が担う

### `terminal.read_line`（Rust側の新規API）

文字列入力（バックスペース対応、カーソル位置管理などの定型的な行編集
処理）は、Lua側で毎回実装する意味が薄い定型処理であるため、Rust側の
薄いユーティリティとして提供する。

呼び出しはLuaの`on_key`処理の中からの**ただの関数呼び出し**であり、
Rustのメインループ構造（`loop { read_key → call on_key }`）自体は
変わらない。`fs.run()`が外部コマンドの実行中はfmのメインループが
一時停止し、終わったら制御が戻ってくるのと同じパターンであり、
既存の設計原則と衝突しない。

> **後日の訂正**: 上記の「Rustのメインループ構造自体は変わらない」という
> 記述は誤りだったことが`fm-interface-design.md`の検討で判明した。
> 「1キー入力＝1回の`on_key`呼び出し」を前提とするスクリーン切り替え方式
> と、「Rustが内部で複数キー入力を監視してから1回だけ返す」`read_line`
> は本質的に噛み合わない。解決策として、`on_key`の**戻り値**で次に
> Rustがどの入力モード（1文字ずつ／行入力）でキーを待つべきかを指定する
> 方式に変更しており、この場合はRustのメインループ自体に「戻り値を見て
> 次の読み取り方を分岐する」という変更が必要になる。詳細は
> `fm-interface-design.md`の該当セクションを参照。

| 引数 | 内容 |
|---|---|
| `x`, `y` | 入力欄の開始位置 |
| `max_width` | 入力を受け付ける最大表示幅（UTF-8表示幅計算を使用） |
| `prompt` | 先頭に固定表示する文字列（`"/"`など） |

| 戻り値 | 内容 |
|---|---|
| `text` | 確定した文字列（キャンセル時は空文字、または`nil`） |
| `confirmed` | `true`(Enter確定) / `false`(Escapeキャンセル) |

### ステップ2: 処理単位に分割

「コマンド実行モード」という枠組みは削除ケースと同様に維持する。ただし
中身の実現方法が異なり、`terminal.read_line`という1回の関数呼び出しで
「追加入力」ステップが完結する点が削除ケースとの違い。

| フェーズ | どのモードで発生するか | 内容 |
|---|---|---|
| 1. コマンド実行のキー操作(`/`) | 通常モード | コマンド実行モードへ遷移 |
| 2. 追加入力 | コマンド実行モード | `terminal.read_line(...)`を呼ぶ |
| 3. コマンド実行 | コマンド実行モード | `read_line`の戻り値でフィルタ処理を実行し、検索状態をセット |
| （通常モードへ復帰） | コマンド実行モード → 通常モード | 通常モードへ遷移。以降のファイル一覧表示は通常モードの`draw`の責務 |

「ファイル一覧表示」はコマンド実行モードの責務ではない。これは削除ケース
でも同様で、削除ケースの「ファイル一覧表示」も本来はコマンド実行モード
自身が描画しているわけではなく、通常モードへ遷移した結果として通常モード
の`draw`が表示するものだった。両ケースの構造がこの点で揃う。

### 削除ケースとの比較（ステップの要否）

| ステップ | 削除 | 検索 |
|---|---|---|
| コマンド実行前の入力受付（確認 or 追加入力） | 必要（`y`/`n`の1文字判定） | 必要（`read_line`による文字列入力） |
| コマンド実行後のメッセージ表示（実行成否表示） | 必要（失敗時のみエラー表示） | 不要（該当なしは一覧が空になることで自明なため） |

「追加入力」自体はどちらのケースにも存在するが、実現方法が異なる
（削除は`on_key`内の1文字判定、検索は`read_line`への委譲）。「実行成否
表示」は削除では必要（失敗があり得るため）だが、検索では結果が一覧の
変化そのもので分かるため不要、という違いがある。

### ステップ3: input / output を定義

「重要な発見」（本ノート末尾参照）を踏まえ、「コマンド実行モードへの
遷移」ではなく「コマンド実行処理の呼び出し」として記述する。

**通常モードの`on_key`が、コマンド実行処理を呼ぶときに渡すデータ
（検索の場合）**

| 項目 | 内容 |
|---|---|
| コマンド名 | `"search"` |
| 入力欄の位置・幅 | `x`, `y`, `max_width` |
| プロンプト文字列 | `"/"` |

**検索コマンドの処理（呼び出しの中で完結する）**

| 処理 | 内容 |
|---|---|
| `read_line`呼び出し | 確定文字列を取得（`read_line`自体が内部でブロッキングして入力を監視する） |
| フィルタ処理実行 | 確定文字列でファイル一覧に検索状態をセット |

**検索コマンドの処理からの戻り値**

- 特になし（検索状態は呼び出しの中で直接データに反映されるため、呼び出し
  元へ返す値は不要）

呼び出しが終わって`on_key`に処理が戻ったら、`draw()`が呼ばれて通常モード
の描画が行われる。削除ケースと同様、「コマンド実行モードが保持する状態」
という概念は不要になる（`read_line`が呼ばれた時点で入力処理はRust側の
`read_line`自身が完結させるため、なおさら保持すべき状態がない）。

## ケース3: 状態変更を伴わない、見た目だけの切り替え（隠しファイル表示のON/OFF）

コマンド実行フロー（基本パターン）を前提に検討する。

題材の検討過程では、ソート順の切り替えや1列⇔2段組の切り替えも候補に
挙がったが、いずれも「アプリケーションの状態（ソートキーや`display_style`
のような値）を変更し、その結果として見た目が変わる」という点で、実質的に
削除・検索と同じ「状態変更を伴う操作」に分類されることが分かった。「状態
変更を伴わない、見た目だけの切り替え」に厳密に当てはまる操作が実在する
かどうかは不明だが、隠しファイル表示のON/OFFを題材として検討を進めた。

### ステップ1: ユーザーから見た処理の流れ

1. **コマンド実行のキー操作**: 通常モードで`.`(ピリオド)を押す
2. **確認 or 追加入力**: なし
3. **コマンド実行**: `show_hidden`の値を反転させる（true⇔false）
4. **実行成否表示**: なし（失敗しようがない）
5. **ファイル一覧表示**: 隠しファイルを含む/含まない一覧が表示される

### 削除・検索との比較（ステップの要否）

| ステップ | 削除 | 検索 | 隠しファイル表示切替 |
|---|---|---|---|
| コマンド実行のキー操作 | `d` | `/` | `.` |
| 確認 or 追加入力 | あり（y/n） | あり（read_line） | なし |
| コマンド実行 | 削除処理 | フィルタ処理 | フラグの反転 |
| 実行成否表示 | あり（失敗時） | なし | なし |
| ファイル一覧表示 | あり | あり | あり |

「確認or追加入力」「実行成否表示」の2ステップが両方とも不要なだけで、
構造自体は削除・検索と同じコマンド実行フローに収まる。

### ステップ2: 処理単位に分割

当初は「確認・追加入力・成否表示が全て不要なら、通常モードの`on_key`が
その場で処理してしまい、コマンド実行モードへの遷移すら省略してよいので
はないか」とも考えたが、削除・検索は「コマンド実行モードに委ねる」のに
隠しファイル切替だけ「通常モードの`on_key`が自分で処理する」という扱いの
不揃いが生じるため、一貫性を優先し、コマンド実行モードへの遷移は行う
方針とする。将来「隠しファイル切替にも確認を挟みたくなった」というような
変更が起きても、コマンド実行モードに統一されていればそこだけ直せばよい。

| フェーズ | どのモードで発生するか | 内容 |
|---|---|---|
| 1. コマンド実行のキー操作(`.`) | 通常モード | コマンド実行モードへ遷移 |
| 2. 確認 or 追加入力 | （スキップ） | 不要なため何もしない |
| 3. コマンド実行 | コマンド実行モード | `show_hidden`を反転 |
| 4. 実行成否表示 | （スキップ） | 不要なため何もしない |
| 5. 通常モードへ復帰 | コマンド実行モード → 通常モード | 通常モードへ遷移。ファイル一覧表示は通常モードの`draw`の責務 |

「コマンド実行モード」に遷移はするが、確認・追加入力・成否表示のステップ
が全て省略されているため、実質的には一瞬でコマンド実行モードに入り、
即座に通常モードへ戻る形になる。これにより、削除・検索・隠しファイル
切替の3ケースがすべて「コマンド実行モードを経由する」という同じ構造で
説明できることが確認できた。

### ステップ3: input / output を定義

「重要な発見」（本ノート末尾参照）を踏まえ、「コマンド実行モードへの
遷移」ではなく「コマンド実行処理の呼び出し」として記述する。

**通常モードの`on_key`が、コマンド実行処理を呼ぶときに渡すデータ
（隠しファイル表示切替の場合）**

| 項目 | 内容 |
|---|---|
| コマンド名 | `"toggle_hidden"` |

（コマンド名以外に渡すデータはない。対象となる状態はグローバルな
`show_hidden`であり、追加の引数は不要）

**隠しファイル表示切替の処理（呼び出しの中で完結する）**

| 処理 | 内容 |
|---|---|
| `show_hidden`の反転 | 現在の値をそのまま反転させる |

**隠しファイル表示切替の処理からの戻り値**

- なし

呼び出しが終わって`on_key`に処理が戻ったら、`draw()`が呼ばれて通常モード
の描画が行われる。3ケースを通じて、「コマンド実行モードが保持する状態」
は一貫して不要（呼び出しのローカルな引数・戻り値、あるいはグローバルな
状態の直接変更で完結する）ことが確認できた。

## ケース4: fmが一時的に制御を手放す操作（外部コマンド実行）

コマンド実行フロー（基本パターン）を前提に検討する。なお、本ケースの
検討はLua⇔Rustの関数呼び出しインタフェース自体の詳細設計は対象外とし、
既存の`fs.run()`のようなAPIを前提とした、fm.lua内部の処理構成のみを
対象とする。

### ステップ1: ユーザーから見た処理の流れ

1. **コマンド実行のキー操作**: 通常モードで`enter`を押す（拡張子に対応
   する`OPENERS`がない、テキスト/バイナリファイルの場合）
2. **確認 or 追加入力**: なし
3. **コマンド実行**: 外部コマンドを実行。この間fmは制御を手放し、外部
   コマンドが画面を専有する
4. **実行成否表示**: なし（外部コマンド自身の画面がその役割を担う）。
   ただし、コマンドによっては終了後に「キー入力を1つ待つ」という
   一時停止を挟みたい場合がある
5. **ファイル一覧表示**: 制御が戻ってきたら、通常モードのファイル一覧を
   再表示

外部コマンド終了後、即座にファイル一覧表示に戻る（パターンA）か、
キー入力を1つ待ってから戻る（パターンB）かは、コマンドの登録ごとに
選択できるようにする方針とする。具体的な設定の記法・デフォルト値は
今は決めない（保留）。この使い分けが必要な理由は、`less`のようなページャ
はユーザー自身が`q`等で抜けるまで表示され続けるため問題にならないが、
実行結果を出力してすぐ終了するコマンド（`grep`など）の場合、即座に
ファイル一覧に戻ると出力を確認する間もなく画面が切り替わってしまう
ため。

### ステップ2: 処理単位に分割

「重要な発見」（本ノート末尾参照）で見出したメインループの構造（`on_key`
の中でコマンド実行処理を呼び、ブロッキングして待ち、戻ってきたら`draw()`
を呼ぶ）に、そのまま当てはめられる。

| フェーズ | 処理 |
|---|---|
| 1. コマンド実行のキー操作(`enter`) | 通常モードの`on_key`が検知 |
| 2〜4. コマンド実行 | `fs.run(command)`を呼ぶ（内部でブロッキング）。`wait_key`設定があれば、続けて`terminal.read_key()`を1回呼んで待つ |
| 5. 通常モードへ復帰 | `on_key`から戻り、`draw()`が呼ばれる |

削除・検索・隠しファイル切替と全く同じ構造（`on_key`の中で呼び出し、
ブロッキングして、戻ってきたら`draw()`）で説明できる。唯一の違いは、
この「コマンド実行」の中身が、fm自身の処理ではなく外部プロセスへの
委譲であるという点。

これで4つの代表ケース（削除・検索・隠しファイル切替・外部コマンド実行）
すべてが、同じ「コマンド実行フロー」の構造で説明できることが確認できた。

### ステップ3: input / output を定義

**通常モードの`on_key`が、コマンド実行処理を呼ぶときに渡すデータ
（外部コマンド実行の場合）**

| 引数 | 内容 |
|---|---|
| コマンド名 | `"run_external"`（あるいは`OPENERS`のキーに応じた名前など、詳細は保留） |
| コマンド文字列 | `fs.run()`に渡すシェルコマンド |
| `wait_key`（コマンド登録ごとの設定） | 実行後にキー入力を1つ待つかどうか |

**外部コマンド実行の処理（呼び出しの中で完結する）**

| 処理 | 内容 |
|---|---|
| `fs.run(command)`呼び出し | 外部コマンドを実行し、fmは制御を手放す |
| `wait_key`が真の場合 | 続けて`terminal.read_key()`を1回呼び、キー入力を待つ |

**外部コマンド実行の処理からの戻り値**

| 項目 | 内容 |
|---|---|
| 終了コード | `fs.run()`の戻り値（現状は特に使っていないが、将来的にエラー判定に使う余地はある） |

呼び出しが終わって`on_key`に処理が戻ったら、`draw()`が呼ばれて通常モード
の描画が行われる。他の3ケースと同様、状態として持続する「モード」は
存在しない。

## 重要な発見: 「モード」は状態ではなく、メインループから呼ばれる処理である

ケース3（隠しファイル表示切替）を検討した後、ケース4に入る前段階として、
「状態変更を伴わない、見た目だけの切り替え」の題材を「1列表示⇔2段組の
切り替え」に広げて検討し直したところ、これまでの「モード」の捉え方に
重要な見直しが必要であることが分かった。

### 発端: 「元の画面に戻らない」ケースの発見

削除・検索・隠しファイル切替の3ケースは、いずれも「コマンド実行モードへ
遷移し、処理が終わったら**同じ通常モードに戻る**」という前提で検討して
きた。

しかし「1列表示⇔2段組の切り替え」を同じ枠組みで考えると、以下のように
コマンド実行モードの前後で**通常モードの描画処理そのものが変わる**ケース
が生じる。

1. ファイル一覧表示を変更するキー入力
2. コマンド実行（1列表示→2段組表示への変更）
3. ファイル一覧表示（1列表示とは異なる、2段組用の描画処理）

これは、削除・検索・隠しファイル切替では起きなかった性質であり、「コマンド
実行モードは必ず元の通常モードに戻る」というスター型の前提にヒビが入る
可能性がある発見だった。

### 解決: モードを状態ではなく処理として捉え直す

この問題は、「モード」を「複数回のメインループ反復にまたがって保持され
続ける状態」として扱うのをやめ、「メインループの中で呼ばれて、仕事を
して、戻ってくるだけの処理（関数）」として捉え直すことで解消する。

具体的なメインループのイメージ（Rust/Luaのインタフェースの詳細は別途
詰める前提の概念図）：

**`draw()`**

1. データ準備
2. **今の状態を見て、どの画面バリエーション（1列表示か2段組か等）を
   描くか決定する**
3. 決定した画面表示処理を呼ぶ
4. return

**`on_key()`**

1. 今の画面表示に対応するキー入力処理を呼ぶ
2. そのキー入力処理が「実行すべきコマンド」を決定する（実行はしない、
   決定するだけ）
3. return
4. コマンド実行処理を呼ぶ（**状態として遷移するのではなく、関数として
   呼ぶ**）
5. コマンドを実行する（内部処理でも外部コマンドでも、この中で完結。
   確認や`read_line`のような追加入力が必要なら、この中でブロッキング
   して待つ）
6. return
7. `draw()`を呼ぶ（再描画）
8. return

これにより、「コマンド実行モードの前後で画面表示が変わる」問題は自然に
解消する。`draw()`はステップ2で**毎回、今の状態を見てどの画面を描くか
決め直す**ため、「元のモードに戻る」のか「別の画面バリエーションに切り
替わる」のかは、`draw()`が呼ばれた瞬間の状態（`display_style`の値など）
を見て、その都度決まるだけになる。「元の場所に戻る」という概念自体が
不要になる。

> **後日の追記**: 上記ステップ5の「確認や`read_line`のような追加入力が
> 必要なら、この中でブロッキングして待つ」という記述は、`fm-interface-
> design.md`の検討でさらに見直されている。確認ダイアログはブロッキング
> ループではなく「別スクリーンへの切り替え」で実現する方式に変わり、
> `read_line`は「`on_key`の戻り値で次の入力モードをRustに指定する」
> 方式へ見直されている。詳細は`fm-interface-design.md`を参照。

### これまでの検討内容への影響

削除・検索・隠しファイル切替・外部コマンド実行の4ケースで「コマンド実行
モードへ遷移する」としていたステップ1・2の記述は、検討過程の記録として
そのまま残すが、ステップ3（input/output）は「コマンド実行処理を呼ぶ」
という理解に合わせて書き直し済み（各ケース参照）。状態としてのモード
管理機構を新設する必要はなく、これは実は現状の`fm.lua`が既に行っている
構造（`move_cursor_down()`や`open_selected()`の最後で`draw()`を呼ぶ）の
延長で実現できる。

### 画面ごとのキー処理: 継承と独立の使い分け（確定事項）

共通部分と差分が混在するキー処理の扱いについて、以下の方針とする。

- **画面バリエーション系**（1列⇔2列表示など、ファイル一覧の見せ方が
  変わるだけのもの）: ユーザーが定義した基本のキー設定を**継承**しつつ、
  必要な部分だけ**上書き・マスク**する
- **ポップアップ系**（PC情報表示など、ファイル一覧とは性質が大きく異なる
  画面）: 継承せず、**一から独立して定義**する

具体的な継承・上書きの実現方法（データ構造、実装方法）は、この方針を
土台にした上で別途検討する。

## コマンド実行処理の入り口を1つにする方針

4ケースの検討では、それぞれ「削除コマンドの処理」「検索コマンドの処理」
のように、コマンドごとに別々の処理を呼ぶ書き方をしていた。しかし、
コマンド実行処理の入り口は1つに統一したい。

そのため、コマンド実行処理を呼ぶ際は、何のコマンドを実行するかを示す
**コマンド名**を、渡すデータに必ず含める（各ケースのステップ3を参照）。
関数名や引数の順序といった具体的なインタフェースの形は今は決めず
（保留）、「呼び出し時にコマンド名を明示する」という点のみを方針として
確定する。
